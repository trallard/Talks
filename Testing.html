Sanity checks across the pipeline

This does not follow a sequential order either but you should always be sanity checking your work.

For example scraping the wrong data, or dropping useful data can seriously compromise your work.

ixek

Basic sanity checks for some steps of the data pipeline

Transformation (data wrangling)

  • Additional or loss of null values after grouping
  • Dataframe dimensions did not deviate from expectations
  • Duplicates during aggregations
  • Math and statistical results from raw and transformed data
  • Coherent data types
  • Schemas deviating from the expected ones
ixek

Analysis

  • Data normalization
  • Residuals and residual plotting
  • Stationary time series
  • Logical order of the operations
  • Shallow vs deep copies of data
  • Never touch your raw data
ixek

Data collection and digestion

  • Consistent schema
  • Serializing ???
  • Garbage in - garbage out

Why bother?

  • Tested code has less bugs
  • Scales well
  • Gives the confidence to iterate quickly
  • Validated data avoid catastrophic failures
  • Reduction of manual testing

Getting started with TDD (test driven development) for data science

Prepare environment

  • Care about tests from the start of your project
  • All jobs should be functions (modularized code)
  • Safe to rerun the job (data exists, push partial results)
  • No hard coded paths
  • Timestamp data
ixek

Unit test locally

  • Test locally each job
  • Overcome challenges with fake data creation
  • Need to identify edge cases
ixek

Input data validation

Bad input will degrade the output and it is likely to pass silently Should:
  • Detect pathological data and fail early
  • Deal with expected data variability
  • Such as missing values, endocing issues, schema changes, data order changes
  • Validate metadata
ixek

Output data validation

Humans make mistakes and these can be easily propagated Should:
  • Check for anomalies
  • Supervised: use a validation dataset and threshold
  • Check for edge cases
